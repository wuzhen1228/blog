念和原理

1.什么是搜索：通过一个关键词或一段描述，得到你想要的结果。

2.如何实现搜索功能：

​ 关系型数据库：性能差，不可靠，结果不准确（相关度低）

3.倒排索引、Lucene和全文检索？

​ 倒排索引的数据结构：

​ 1.包含这个关键词的document list

​ 2.关键词在每个doc中出现的次数

​ 3.关键词在整个索引中出现的次数

​ 4.关键词在当前doc中出现的次数

​ 5.每个doc的长度，越长相关度越低

​ 6.包含这个关键词的所有doc的平均长度

​ Lucene： jar包，帮我们创建倒排索引，提供了复杂的API。

​ 如果Lucene做集群实现搜索：

​ 1.节点一旦宕机，数据丢失。

​ 2.自己维护。（自己创建管理索引）单台节点的承载请求的能力是有限的，需要人工做负载。

4.Elasticsearch：分布式、高性能、高可用、可伸缩、易维护。

​ 分布式的搜索和数据分析引擎：

​ 优点：

​ 1.面向开发者友好，屏蔽了Lucene的复杂特性，集群自动发现（cluster discovery）

​ 2.自动维护数据在多个节点上建立

​ 3.会帮我做搜索请求的负载均衡

​ 4.自动维护冗余副本，保证了部分节点宕机的情况下仍然不会有任何数据丢失

​ 5.基于Lucene提供了很多高级功能：符合查询、聚合分析、基于地理位置。

​ 6.对于大公司可以构建几百台服务器的大型分布式集群，处理PB级别数据。

​ 7.相对于传统数据库，提供了全文检索，同义词处理，相关度排名，聚合分析以及海量数据的近实时处理，这些传统数据库做不到。

​ 应用领域：

​ 1.百度（全文检索、高亮、搜索推荐）

​ 2.各大网站的用户行为日志（用户点击、浏览、收藏、评论）

​ 3. BI系统，数据分析，数据挖掘统计

​ 4.Github：代码托管凭证，几千亿行代码

​ 5.ELK：（Elasticsearch（数据存储）Logstash（日志采集）Kibana（可视化））

5.ES核心概念：

​ 1.Cluster：集群，每个集群至少包含两个节点。

​ 2.Node：集群中的每个节点，一个节点不代表一台服务器。

​ 3.Field：一个数据字段，与index和type一起，可以定位一个doc

​ 4.Document： ES最小的数据单元 Json格式

​ 5.Type ： 逻辑上的数据分类

​ 6.Index ：一类相同或者类型的doc，比如一个员工索引，商品索引

Shard分片：

1. 一个index包含多个shard，默认5P，默认每个P分配一个R，P的数量在创建索引的时候设置，如果想修改，需要重建索引。

   2. 每个Shard都是一个Lucene实例，有完整的创建索引的处理请求能力。
   3. ES会自动为nodes上为我们做shard均衡。
   4. 一个doc是不可能同时存在多个Pshard中的，但是可以存在多个RShard中。
   5. P和对应的R不能同时存在同一个节点。所以最低的可用配置是两台节点，互为主备。

kafka介绍

提供了消息的订阅与发布的消息队列，用作系统间解耦、异步通信、削峰填谷等作用

常见的消息队列工作模式分为两种

一：

至多一次，消息生产者将数据写入消息系统，然后由消费者负责去拉消息服务器中的消息，一旦消息被确认消费后，由消息服务器主动删除队列中的数据，这种消费方式一般只允许被一个消费者消费，并且消息队列中的数据不允许被重复消费。

二：

没有限制，同上诉消费形式不同，生产者发布完数据以后，该消息可以被多个消费者同时消费，并且同一个消费者可以多次消费消息服务器中的同一个记录，主要是因为消息服务器一般可以长时间存储海量消息

kafka集群

是以Topic形式负责分类集群中的Record，没一个Record属于一个Topic，每个Topic底层都会对应一族分区的日志用于持久化Topic中的Record。

分区

生产者把key进行hash运算，把消息均匀的分散到不同分区，可以把相同的key的消息发送到相同的分区

每个Topic都有一个Broker，负责存储Topic的分区数据，每个Topic会有多个副本（备份），多个Broker会根据设置的副本数量，互相备份。

如果集群中的某个Broker宕机了，集群中的剩余Broker进行选举。

生产者&消费者

消费者对单个分区也是分别独立的，每个消费者会有自己的消费偏移量（序号）

消费者使用Consumer Group名称标记自己，Topic中的每条记录都会传递到每个Consumer Group中的一个消费者实例，如果所有消费者都具有相同的ConsumerGroup。

分区会根据消费组和组内的消费者实例进行均分

如果消费者实例有个挂掉了。那这时候如果有空闲实例，则会连接空闲的消费者实例

高性能-顺序写&mmap

高吞吐量，所有的消息是以日志是以保存或者缓存在磁盘上的，可以支持百万级的写入请求

为了优化写入速度，采用了技术顺序写入和MMFile。

技术顺序写入：因为硬盘是机械结构，每次读写都会寻址->写入，寻址的动作是非常耗时的，如果采用顺序I/O，会省去大量的内存开销和寻址时间，但kafka的数据并不是实时的写入硬盘中的。

利用OS中的Pagecache提高IO效率。MMF是直接调用的一个进程和内核共享的内存区域，直接把数据写到这个内存区域，然后由OS去把内存中的数据持久化到磁盘。

Redis持久化

发表于2020-02-23更新于2020-02-23

Redis

RedisDB持久化

时点性的进行快照

使用内核的fork(copy on write)指令，创建一个子进程，子进程执行持久化到db.file，这样做的原因是父子进程对数据的修改，互相都看不到，是因为key是存在虚拟内存中，但value是放在物理内存中的，如果主进程在子进程备份过程中修改了value，主进程的key会指向新的物理内存中的值，但子进程还是指向原来的value。

以上说的是bgsave，redis是可以正常使用的。

redis还提供了普通save，会导致redis不可用，这种情况下就是关机维护之类的，确定不用了。就用这个save。

AOF

因为DB还是会丢失数据，所以有了AOF的技术。

redis的写操作记录到文件中，如果rdb和aof同时开启，只会用aof恢复。

redis4.0之后，因为全量恢复aof日志，会慢，所以会先把时间节点最近的rdb重写到aof日志中，再把增量的数据append到aof中。

Redis缓存击穿穿透雪崩

发表于2020-02-21更新于2020-02-21

Redis三大问题

缓存击穿：

  1. key有过期时间
  2. 开启LRU或者LFU

如果key过期了，然后有大量流量突然访问这个key，流量就会访问数据库。

解决方案：

1.    阻止并发到达DB

   2.    所有请求都会get key，所有都get到空之后，使用setnx加锁，锁要设置过期时间，如1秒。
      3.    如果获取锁成功了，那就去DB重新取数据，在set到key中
         4.    如果失败了，就随机睡眠1秒以内，然后再第2个步骤
            5.    如果第三步去DB拉数据失败了，就启用多线程，主线程去获取DB数据，子线程（如0.5秒）去监控是否拉取成功，没成功就更新锁的过期时间。

缓存穿透：

从业务层接收查询的是系统里根本不存在的数据。

解决方案：使用布隆过滤器。

问题：布隆过滤器只能增加，不能删除，更换过滤器，布谷鸟，或者布隆过滤器+

缓存雪崩：

大量Key同时失效，简介造成大量的访问到达DB。

解决方案：使用随机过期时间，在原过期时间上随机加几分钟，再使用缓存击穿的方案，慢慢过渡。

分布式锁：

  1. setnx
  2. 过期时间
  3. 多线程（守护线程） 定时增加过期时间

线程池

Callable和Runnable的区别，Callable有返回值。

Future

存储执行的将来才会产生的结果。

FutureTask

Future+Runnable 又可以执行，又可以存结果。

CompletableFuture

底层的实现的是ForkJoinPool。管理任务，对任务进行组合。支持链式的表达式。

ThreadPoolExecutor

corePoolSize：核心线程数，永久存活

maximumPoolSize：最大线程数，如果没活干了，等待生存时间结束，还没活干，就归还给操作系统。

keepAliveTime：生存时间

TimeUnit：生存时间的单位

BlockingQueue：任务队列，比如指定ArrayBlockingQueue等BlockingQueue的实现类。

ThreadFactory：线程工厂，产生线程的工厂类，可自己实现ThreadFactory接口，对线程命名等等设置。

RejectStrategy：拒绝策略，线程池忙而任务队列满了，要执行的拒绝策略。一般会自定义这个策略，比如订单，肯定要把订单任务放到中间件或者数据库，记录日志。

​ jdk自带的拒绝策略：

​ abort：抛异常

​ Discard：扔掉。不抛异常

​ DiscardOldest：扔掉排队最长的任务。

​ CallerRunsPolicy：交给调用者执行，比如main要加进来，就main线程自己执行

源码

SingleThreadExecutor

线程池只有一个线程，可以保证线程是顺序执行的。

CachedPool

进入到这个线程池的线程，会立即执行。

FixedThreadPool

固定的线程数量，核心线程数与最大线程数一致。

ScheduledThreadPool

定时任务线程池。

WorkStealingPool

ForkJoinPool

把大任务切分成小任务运行。

ParalleStream

并行流处理，比单一线程的Steam处理更快，底层实现是ForkJoinPool。

用ThreadLocal解释弱引用：

​ Thread类中有个属性叫threadLocals，threadLocals的实现类是ThreadLocalMap，这个map中的key存放的key就是ThreadLocal，但这个ThreadLocal是继承WeakReference（弱引用），所以他key会在Thread=null的时候，被回收。

如果使用强引用，会发生内存泄漏。ThreadLocal永远不会被回收。

但弱引用也会有内存泄漏的风险存在，ThreadLocal被回收后，key就会变成null，但其value还会指向原来的内存地址，key变成null也会导致value不能访问，因此还会有内存泄漏问题。

所以在ThreadLocal使用完后，调用其remove方法。

容器分为两类，Collection和Map。

Collection

List

vector：自带锁，基本不用。

ArrayList：不是线程安全的。是动态数组，对除尾部元素的数据插入效率低，要移动其他元素下标。

LinkedList：是链表，获取元素效率低，得遍历。

CopyOnWriteList：写实复制，就是写的时候加锁，读的时候不加锁。

Set

Queue

ConcurrentLinkedQueue:内部使用CAS实现锁。

LinkedBlockingQueue：无界阻塞队列

ArrayBlockingQueue：：有界阻塞队列

DelayQueue：时间排序，按时间进行任务调度，创建线程的时候参数里加个时间戳，到这时间线程就运行。

SynchronusQueue：容量为0，两个线程中传递任务。

TransfeQueue:多一个transfer方法，与blocking的put方法类似，put方法是把数据存了。就继续运行，transfer是必须有其他线程或者这个数据被处理了，才继续运行，否则一直阻塞。

BlockingQueue：用put方法和take方法实现了生产者消费者问题，没有数据线程会阻塞住。

PriorityQueue：内部实现了一个最小堆，内置排序。

Map

HashTable：自带锁，基本不用。HashMap也有带锁的版本，在Collection的子类实现中，synchornizeMap，区别是锁的粒度会小一些，影响不大。

效率：写效率比较高，读效率很低。

HashMap：

效率：写效率比较高，读效率很低。

ConcurrentHashMap:

效率：写效率比较低，读效率超高。内部使用CAS操作。

ConcurrentSkipListMap：高并发且有排序。

TreeMap：红黑树。有排序。
