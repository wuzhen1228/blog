Mysql框架

客户端 -> mysql server -> 连接器（控制用户的连接） -> 分析器（词法分析、语法分析） -> 优化器（优化SQL语句，规定流程执行） -> 执行器（SQL语句的实际执行组件）-> 存储引擎（不同的存放位置，不同的文件格式）

客户端 ->连接器：采用数据库连接池，减少频繁的开关连接；

优化器：可以查看SQL语句的执行计划，可以采用对应的优化点，来加快查询；

存储引擎：innoDB（磁盘），MyISAM（磁盘），memory（内存）；

索引文件

不同的存储引擎，数据文件和索引文件存放的位置是不同的，因此有了分类：

​ 聚簇索引：数据和文件放在一起，innoDB；

​ .frm：存放的是表结构；

​ .ibd：存放数据文件和索引文件；

​ mysql的innoDB存储引擎默认情况会把所有的数据文件放到表空间中，不会为没一个单独的表存放一份数据文件，如果需要把每个表单独使用文件保存，设置如下属性：

​ set global innodb_file_per_table = on;

​ 非聚簇索引：数据和索引单独一个文件，MyISAM；

​ .frm：存放表结构；

​ .MYI：存放索引数据；

​ .MYD：存放实际数据；

Mysql事务隔离测试

1.测试事物隔离性，要先关闭mysql的自动提交。

2.要打开两个命令窗口，分别关闭自动提交和修改事物隔离机制。

3.准备下数据

4.事务隔离级别说明

隔离级别 当前级别下可发生的异常情况
read uncommitted–读未提交 脏读，不可重复读，幻读
read commited–读已提交 不可重复读，幻读
repeatable read–可重复读 幻读
seariable–序列化执行

5.测试脏读，以下简称窗口A、B，分别进行读写测试

​ 分别执行后发现，B窗口读到了A窗口修改的数据，但是A窗口并没有执行commit操作，所以B读到了不存在的数据，这种情况就是发生脏读。

​ A窗口执行ROLLBACK后，B窗口数据也会回滚；但A窗口commit后，数据无法回滚，数据已经被永久性修改。

6.测试不可重复读

​ 执行到这里发现A窗口的数据是修改后的数据，B窗口的还是老王，脏读的情况没有了。

​ 然后A窗口commit后。B窗口也能读取到A窗口修改后的数据。

​ 结果就是同一个事务中多次读取数据，会出现不一致的情况。

7.测试幻读

​ 这里A窗口插入一行数据，并且提交事务。在B窗口查询person表，看不到A窗口插入的数据

​ B窗口再插入数据，会提示ID为4的数据已存在，主键不能重复。这种就是出现幻读现象，A窗口的数据在插入并提交后，已经永久保存到本地磁盘了，B窗口还是原始数据。

8.结语

​ 隔离级别升到seariable后，以上情况都不会再出现。

Mysql执行计划

Mysql 执行计划

explain

id:

​ 复杂语句中可能有多个id号。表示select子句的执行顺序。id号越大越先执行，id号相同，从上往下按序执行。

select_type:

​ 主要用来分辨查询类型，是普通查询还是联合查询或者是子查询

SIMPLE:

​ 表示简单的语句。

PRIMARY:

​ 查询中如果包含了任何复杂的子查询，最外层查询则被标记为primary。

UNION：

​ 若第二个select出现在union后，则被标记为union。

DEPENDENT UNION:

​ 跟union类似，此处的depentent表示union或union all联合而成的结果会受外部表影响。

UNION RESULT:

​ 从union表中或者结果的select。

SUBQUERY:

​ 在select或where列表中包含子查询。

DEPENDENT SUBQUERY:

​ SUBQUERY的子查询受到外部表查询的影响。

DERIVED:

​ from子句出现的子查询，也叫派生类。

UNCACHEABLE SUBQUERY:

​ 表示子查询的结果不能被缓存。

UNCACHEABLE UNION:

​ 表示union的结果不能被缓存。

table:

​ 对应行正在访问哪个表，表名或者列名，可能是临时表或者union合并结果集。

​ 1.如果是具体的表名，则表名从实际的物理表中获取数据，也可以是表的别名。

​ 2.表名是derivenN的形式，表示使用了id为N的查询产生的衍生表。

​ 3.当有uion result 的时候，表名是union n1,n2等的形式，n1,n2表示参与union 的id。

type:

​ type显示的是访问类型，访问类型表示我是以何种方式去访问我们的数据，最容易想的是全表扫描，直接暴力的遍历一张表去寻找需要的数据，效率非常低下，访问的类型有很多，效率是从最好到最坏依次是：

system>const>eq_ref>fulltext>ref_or_null>index_merge>unique_subquery>index_subquery>range>index>all

​ all:全表扫描，一般情况下出现这样的sql语句，而且数据量比较大的话那就需要优化了。

​ index:全索引扫描。比all效率高，主要有两种情况，一种是当前的查询是覆盖索引，即我们需要的数据在索引中可以获取，或者是使用了索引进行排序，避免数据的重排序。

​ range：表示利用索引查询的时候限制了范围，在指定范围内进行查询，这样避免了index的全索引扫描，适用的操作符：=，<>,>,>=,<,<=,is null,between,like,or in()

​ index_subquery:利用索引来关联子查询，不用再扫描全表。

​ unique_subquery:连接类型与index_subquery一样，只不过子查询关联的是唯一索引。

​ ref_or_null：对于某个字段需要关联条件，也需要null值的情况下，查询优化器会选择这种查询方式。

​ ref：使用了非唯一性索引进行数据的查找。

​ eq_ref：使用唯一索引进行查找。

​ const：这个表至多有一个匹配行。

​ system:表里只有一行记录（等于系统表），这是const的特例，平时不会出现。

possible_keys:

​ 显示可能应用在这张表中的索引，一个或多个，查询涉及到的字段上若存在索引，则该索引会被列出，但不一定被查询实际使用。

key:

​ 实际使用的索引，如果为null，则没有使用索引，查询中使用了覆盖索引，则该索引和查询的select字段重叠。

keylen:

​ 表示索引中使用的字节数，可以通过key_len计算查询中使用的索引长度，在不损失净度的情况下长度越短越好。

ref:

​ 显示索引的哪一列被使用了，如果可能的话，是一个常数。

rows:

​ 根据表中的统计信息及索引的使用情况大致估算出找出所需记录需要读取的行数，此参数很重要，直接反应的sql找出了多少数据，在完成目的的情况下越少越好。

filtered:

Extra:

​ 包含额外的信息。

​ using filesort：说明mysql不能根据索引进行排序，只能通过排序算法进行排序，会消耗额外的位置。

​ using temporary：建立临时表来保存中间结果，查询完成之后临时表删除。

​ using index：这个表示当前的查询时覆盖索引的，直接从索引中读取数据，而不用访问数据表，如果同时出现using where 表明索引被用来执行索引键值的查找，如果没有，表明索引被用来读取数据，而不是真正的查找。

​ using where : 使用where 条件过滤。

​ using join buffer: 使用连接缓存。

​ impossible where :where语句的结果总是false。

数据库设计三范式

三范式的主要作用是减少数据库中数据冗余。

第一范式：确认每列保持原子性，例如收货地址，把省、市、区拆分成三个字段，分别保存相应数据。

现在有一个地址字段：北京市丰台区荣盛小区。对其进行拆分

第二范式：确保表中每列和主键相关，列必须直接依赖主键。

第三范式：需要确保表中每列数据都和主键直接相关，而不是间接相关。就是说表里面的列不能出现其他表的非主键字段

kafka介绍

提供了消息的订阅与发布的消息队列，用作系统间解耦、异步通信、削峰填谷等作用

常见的消息队列工作模式分为两种

一：

至多一次，消息生产者将数据写入消息系统，然后由消费者负责去拉消息服务器中的消息，一旦消息被确认消费后，由消息服务器主动删除队列中的数据，这种消费方式一般只允许被一个消费者消费，并且消息队列中的数据不允许被重复消费。

二：

没有限制，同上诉消费形式不同，生产者发布完数据以后，该消息可以被多个消费者同时消费，并且同一个消费者可以多次消费消息服务器中的同一个记录，主要是因为消息服务器一般可以长时间存储海量消息

kafka集群

是以Topic形式负责分类集群中的Record，没一个Record属于一个Topic，每个Topic底层都会对应一族分区的日志用于持久化Topic中的Record。

分区

生产者把key进行hash运算，把消息均匀的分散到不同分区，可以把相同的key的消息发送到相同的分区

每个Topic都有一个Broker，负责存储Topic的分区数据，每个Topic会有多个副本（备份），多个Broker会根据设置的副本数量，互相备份。

如果集群中的某个Broker宕机了，集群中的剩余Broker进行选举。

生产者&消费者

消费者对单个分区也是分别独立的，每个消费者会有自己的消费偏移量（序号）

消费者使用Consumer Group名称标记自己，Topic中的每条记录都会传递到每个Consumer Group中的一个消费者实例，如果所有消费者都具有相同的ConsumerGroup。

分区会根据消费组和组内的消费者实例进行均分

如果消费者实例有个挂掉了。那这时候如果有空闲实例，则会连接空闲的消费者实例

高性能-顺序写&mmap

高吞吐量，所有的消息是以日志是以保存或者缓存在磁盘上的，可以支持百万级的写入请求

为了优化写入速度，采用了技术顺序写入和MMFile。

技术顺序写入：因为硬盘是机械结构，每次读写都会寻址->写入，寻址的动作是非常耗时的，如果采用顺序I/O，会省去大量的内存开销和寻址时间，但kafka的数据并不是实时的写入硬盘中的。

利用OS中的Pagecache提高IO效率。MMF是直接调用的一个进程和内核共享的内存区域，直接把数据写到这个内存区域，然后由OS去把内存中的数据持久化到磁盘。
